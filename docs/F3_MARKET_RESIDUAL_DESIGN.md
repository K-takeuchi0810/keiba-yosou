# F3 市場残差特徴 設計 (2026-07-03 確定版)

エッジ探索の本命フェーズ。設計制約 4 件 (封印ホールドアウト / mining除外 ablation /
review≠外部検証 / PIT T−n コード強制) はユーザ合意済み (2026-07-03) で本書の前提。

## 0. 在庫実測 (2026-07-03) — 設計を規定する事実

| 事実 | 実測値 | 含意 |
|---|---|---|
| 歴史 PIT スナップ (data/raw/0B31) | **600 レース / 18 開催日** (2026-05-02〜06-28) | 開発に足る量がない |
| スナップ数/レース | 216 レースが 1 枚のみ、大半 1-8 枚 | ドリフト (≥2点) を計算できないレースが多数 |
| 最終スナップの時刻 | **50% が発走後** (旧 mining バグ期) | 旧在庫の相当部分は T−n ゲートで落ちる |
| 収集系の現状 | `fetch_fresh_odds --window 25` = T−25〜lead 分のみ、直近稼働率 100% | **朝の基準オッズを収集していない** → 「朝→直前ドリフト」が作れない |
| DB 格納 | horse_races は最新 1 スナップのみ (UPDATE) | **時系列テーブルが無い** |
| 2025 以前 | PIT スナップ皆無 (確定オッズのみ) | 過去データでの F3 開発は不可能 |

**結論: F3 は「過去データで作って検証」できない。前向き蓄積が主軸**であり、
数ヶ月の蓄積待ちが構造的に必要 (制約(1) の含意と一致)。

## 1. フェーズ構成

### F3-a: 計測インフラ (即着手可、~1-2 セッション)
1. **odds_snapshots テーブル新設** — レース×馬×取得時刻の時系列。
   PK (race複合キー, horse_num, fetched_at)。horse_races の単一スナップ UPDATE は現行維持
   (GUI 互換)、INSERT は追加経路。
2. **raw 0B31 のバックフィル** — ファイル名 `0B31_<racekey>_<epoch>.jvd` から 600 レース分の
   時系列を odds_snapshots へ復元 (少量だが動作検証・初期分析用)。
3. **収集 cadence の拡張** — 朝基準スナップ (開催日 9:30 前後に全レース 1 回) を scheduler に追加。
   既存の T−25〜lead 窓収集はそのまま。→ ドリフト = 朝基準 → T−n の 2 点が全レースで揃う。
4. **PIT ゲートのコード強制** (制約(4)、review 降格宣言(2)):
   - 特徴計算が使ってよいスナップ = `fetched_at IS NOT NULL` **かつ** `fetched_at ≤ 発走時刻 − n 分`。
   - NULL fetched_at (=確定・発走後) を市場特徴に使う経路はコードで遮断 + ガードテスト。
   - **n の値は config 一元管理** (`PIT_GATE_MINUTES`)、backtest と live で同一値を強制。
5. **coverage 監視拡張** — 朝基準スナップの取得率を fresh_odds_coverage に追加。

### F3-b: 蓄積 + 開発 (dev 窓)
- **dev 窓 = 2026-07-04 〜 封印開始日の前日**。この間の蓄積データで特徴設計・イテレーション自由。
- 特徴ファミリ (候補、dev で選抜):
  - 単勝オッズドリフト: log(朝オッズ / T−n オッズ)、市場 implied prob の変化量
  - モデル順位 vs 市場順位の乖離 (rank divergence)、乖離の時間変化
  - overround (合成オッズ) の変化、上位人気への資金集中度の変化
- 判定は dev 窓内の walk-forward のみで行い、封印データには一切触れない。

### F3-c: 封印判定 (一度だけ)
- **封印開始日**: 事前宣言し、その日以降の蓄積は判定まで参照禁止 (分析スクリプトも実行しない)。
- **判定基準 (事前宣言)**:
  1. 主指標: F3 特徴込みモデルの buy 判断について封印窓 OOS 回収率 **CI 下限 > 100%** →実弾候補昇格。
     CI 下限 ≤100% でも点推定が dev 窓と整合して >100% なら「紙運用」検討、それ未満は棄却。
  2. **mining 除外 ablation** (制約(2)): mining 特徴を抜いた構成でも F3 の寄与 (ΔAUC/Δ回収率) が
     正に残ること。残らなければ「F3 は独立情報ではない」と判定。
  3. 判定は**一度だけ**。失敗時は F3 を棄却し、封印窓は次仮説の dev 窓に転用可 (その時点で封印解除)。
- **判定時期**: 封印窓に最低サンプル (下記ユーザ決定) が貯まった時点。

## 2. ユーザ決定事項 (2026-07-03 確定)

| # | 決定 | **確定値** |
|---|---|---|
| D1 | **T−n の n** | **n=10 分** (`PIT_GATE_MINUTES=10`)。T−10 より後のスナップは特徴に使用禁止。backtest と live で同一値を config 一元管理 |
| D2 | **封印設計** | **dev = 2026-07-04〜09-30 / 封印 = 2026-10-01〜 / 判定 = 封印窓 ≥800 レース到達時に一度だけ** (~12月上旬見込み)。封印窓のデータには判定まで分析スクリプトも実行しない |

## 3. リスクと正直な期待値

- ドリフト特徴の文献的エビデンスは「smart money は遅く動く」系だが、JRA の控除率 20-30% を
  超えるかは未証明 (deep-research 済)。**F3 も棄却される可能性は十分ある** — その場合も
  「市場残差は取れない」という高価値の知見が残る (判定プロトコルがあるから言い切れる)。
- 収集系が単一障害点 (PC 稼働・scheduler)。朝スナップ欠損日は dev/判定から除外するルールを
  coverage で機械化。
- mining 依存 67% の監視 (weekly_monitor) は F3 と独立に継続。
