# 採点ルーブリック (全専門家共通)

各専門家はこの 5 段階基準で採点する。**項目別スコア + 総合スコア (= 項目平均、小数 1 桁まで)** を必ず出す。

## 5 段階の意味

| 点 | レベル | 意味 |
|---|---|---|
| **5** | 模範的 | 同種アプリの中でも上位。改善余地ほぼなし |
| **4** | 良好 | 軽微な改善余地はあるが現状で運用可能 |
| **3** | 標準的 | 目立つ問題は無いが優位性も無い |
| **2** | 改善必要 | 運用に支障が出るレベル |
| **1** | 致命的 | 即修正が必要 |

迷ったら **辛口側** に寄せる (アプリの質を上げるための採点なので甘い 4 は意味がない)。

## 出力フォーマット (必ずこの形式で返す)

```markdown
# <専門家名> 採点

## 総合: X.X / 5

## 項目別

- **<項目1>: N/5** — 1〜2 行の所見
- **<項目2>: N/5** — 1〜2 行の所見
- **<項目3>: N/5** — 1〜2 行の所見
- **<項目4>: N/5** — 1〜2 行の所見
- **<項目5>: N/5** — 1〜2 行の所見

## 主な改善提案 (優先 1 件、最大 3 件)

1. **<タイトル>** — 何を変えるか / 期待効果 (2〜3 行)
2. ...

## 前回からの差分 (前回スコアがあれば)

- 項目1: 3 → 4 (+1) 改善: <理由>
- 項目2: 4 → 4 (±0) 維持
- ...
```

## ルール

1. **対象ファイルだけを読む**。専門範囲外のファイルを読み込んでトークンを浪費しない
2. **「現状が標準的なら 3」**。下駄を履かせない
3. **改善提案は具体的に**。「もっと良く」「整理する」のような抽象論は禁止。「`gui/app.py:1234` の N を M に置換」レベルの粒度
4. **過去 scorecard を確認**: `data/scorecards/` に過去ファイルがあれば読み、差分を出す
