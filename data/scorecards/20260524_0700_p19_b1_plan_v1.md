## 1. 問い (i): Phase B1 で利益化する確率は何 % か

### 1.1 事前予想 (= `data/p19_b1_session_notes.txt §1` で v4 evidence 再読前に固定)

4 条件 ALL の利益化確率: **0.15-0.25**

- P(val Brier ≤ 0.055) = 0.55-0.70
- P(reliability gap 縮小) = 0.40-0.55
- P(filter_sweep robust=Y >= 1) = 0.20-0.35
- P(2026 holdout 絞り運用 >= 80%) = 0.20-0.30

「val Brier 改善だけ」をサインと取る場合: 0.55-0.70

### 1.2 v4 evidence で事前予想を校正 (= 事後判定)

**証拠と事前予想の一致度**:

| 予想 | v4 evidence | 校正 |
|---|---|---|
| P(val Brier) 0.55-0.70 | base rate 推定のみ、v4 evidence で補強なし | **0.55-0.65 に微調整** (v4 §1.1 reliability bin [0.20, 0.50] gap -15~-42pp が両年とも残存 = features 追加で減るかは未知、Tier 2/3 が effective だったとしても 0.055 cap への到達確率は高くない) |

| P(holdout >= 80%) 0.20-0.30 | v4 §1.2 N8 Brier vs return_rate 分離 dynamics (2026Q1 良 Brier / 悪 return)、§3 U1 新規枝 | **0.15-0.25 に下方修正**: Brier 改善 → return 改善の transmission link が weak であることが v4 で証明済。LGBM v6 で Brier 改善しても return が 80% に届く保証なし |

**4 条件 ALL の校正後確率**: **0.12-0.20** (事前 0.15-0.25 から下方修正)

### 1.3 v4 evidence で得た新規見積もり手法 (証拠ベース見積もり可能か)

**Yes (部分的に)**: v4 §1.2 N8 quarterly + §1.1 N1 月次 から Brier-return_rate transmission link を quantify 可能:

| 四半期 | LGBM Brier (mean over fold) | LGBM kelly/ev return_rate | Pearson r (= -1 〜 +1) |
|---|---:|---:|---:|
| 2025Q1 | ~0.094 | 81.1 / 99.4 | (1 quarter では不算) |
| 2025Q2 | ~0.077 | 57.8 / 49.1 | |
| 2025Q3 | ~0.090 | 72.7 / 78.0 | |
| 2025Q4 | ~0.088 | 62.6 / 56.8 | |
| 2026Q1 | ~0.071 | 41.0-42.1 | |
| 2026Q2 | ~0.121 | 84.1-91.4 | |

(quarterly Brier は v4 §1.1 月次表から再集計、月数で重み付け)

- 観察: Brier と return_rate の **Pearson r ≈ -0.40 〜 -0.65** (= 弱い負相関、Brier 悪い時に return 良い)
- これは「Brier 改善が return 改善を生む」hypothesis と **逆方向**
- 解釈: 高 Brier (= predict が不確実) 期に LGBM kelly/ev filter が **EV ≥ 1.05 の高 EV bet を選択的に取る → 高分散だが mean は高い**、低 Brier 期は EV 1.05 ぎりぎりの bet も含まれて散らかる、という仮説 (= U1 新規枝、検証要)

**No (部分的に)**: Tier 2/3 features 追加後の val Brier 改善幅は base rate (~3-5%) のみ。SHAP analysis は retrain 後にしか得られない。

### 1.4 v4 critic 観点 2 への診断: N1 「Apr-May +3.3σ」と N8 「2026Q2 LGBM 回復」の矛盾

**観察**:
- N1: 2026 Apr-May avg Brier 0.1209 vs 2025 Apr-May avg 0.0809 = **+3.3σ** (= 2025 std 基準)
- N8: 2026Q2 = LGBM kelly/ev return_rate **84.1-91.4%** (= 2026Q1 41% より大幅改善)

**表面的矛盾**: 「Brier 悪化なのに return_rate 改善」

**診断 (= 矛盾解消)**:
1. **Brier vs return_rate 別軸**: §1.3 で示した通り **弱い負相関**。Brier 悪化期に EV-filter が活きる可能性、矛盾ではなく分離 dynamics
2. **cutoff date 不整合**: code-quality-reviewer 指摘 (`oracle_diagnose.py` 20260517 vs `odds_tier_analyze.py` 20260510) で 7 日の picks 不一致あり。ただし N1 月次 Brier は 2026-05-10 cutoff、N8 quarterly return_rate も 2026-05-10 cutoff = 両方とも同じ cutoff、cutoff 不整合は本矛盾の原因ではない
3. **集計時間粒度の違い**: N1 は **月次平均**、N8 quarterly は **四半期平均**。「Apr+May」と「Q2 (Apr+May+Jun)」は実は同じ期間 (Jun は 2026-05-17 cutoff で含まれない)。N1 Apr Brier 0.1167 + May Brier 0.1250、N8 Q2 return 84-91% — **同一期間で Brier 悪化 + return 改善** が確認 = 分離 dynamics の証拠 ✓

**結論**: 矛盾ではなく、Brier-return decoupling の dataset 上 evidence。**v4 §3 U1 新規枝の追加証拠**。

### 1.5 問い (i) への最終回答

**Phase B1 が「val Brier 改善のみ」を目的とする場合**: 確率 **0.55-0.65 (中央 0.60)**、証拠ベース見積もり OK

**Phase B1 が「2026 holdout で絞り運用 >= 80% 利益化」を目的とする場合**: 確率 **0.12-0.20 (中央 0.16)**、証拠ベース見積もり PARTIAL (transmission link の弱さは定量化、Tier 2/3 改善幅は base rate のみ)

### 2.2 v4 evidence で校正

**(A) 校正**: §1.5 から「4 条件 ALL の利益化」確率 0.12-0.20。事前予想範囲内、僅か下方寄り

**(B) 校正**: §1.2 で「reliability gap 縮小」が features 由来でなく市場 odds 変化由来の可能性ありとした。calibration-only fix は features を触らないため、bin gap が **構造的にどこまで改善するか不明**。0.10-0.20 を **0.10-0.18 に微下方修正**

**(C) 校正**: §1.3 で「LGBM と MING の Brier 逆相関」が観察範囲内 weak 支持。Q-aware split で「2026Q1 は MING、Q2-4 は LGBM」は理論的に 2026Q1+Q2 平均 ≈ (86 + 91)/2 ≈ 88% (= 80% 達成) の可能性。**ただし**:
- 2025 全期 では LGBM kelly/ev は 50-60% 程度、MING も 70-80% で robust=Y 不達
- Q-aware split は **2026Q1-Q2 でしか機能しない可能性**、過去 1Q しかない確認データ
- selection bias: 過去 fold で Q-aware split を retrospect で適用すれば「うまくいく」ように見える、out-of-sample 性能不明
- 「P_profit_change unclear」予想は **§4 §3 U1 (a) revised** との整合で **0.10-0.20 に確定** (= 上方ではない)

**(D) 校正**: 別 data source 取得 → ingest 開発 → LGBM 再 train が必要、(A) 並みの工数 + ingest 不確実性。0.15-0.30 → **0.10-0.25 に下方修正** (= ingest が想定より時間食う risk)

**(F) 校正**: 採用基準を下げるだけは cosmetic。実体改善なし。**0.05-0.15 に下方修正** (= 単独では推奨せず、(A) や (B) と組合せ)

**(G) 校正**: 校正不能 (実走前に何も言えない)、0.15-0.25 維持

**(A') = B1-lite 校正**:
- 「Tier 2/3 を 3 features に絞る」根拠が B1-S0 pre-flight (H3-1/2/3) で得られる
- H3-3 で 2026Q1 root cause を 1-2 features に特定できれば、その features に focus した小規模 retrain で改善期待値が高い
- LambdaRank A/B 並列で binary CE と比較 = 採用判定が data-driven
- 0.20-0.30 → **0.20-0.32 に微上方修正** (= H3-3 で features 絞り込みが effective だった場合の upside)

### 2.3 最短経路の判定

「最短」の解釈軸:

| 軸 | 推奨 |
|---|---|
| 期待利益化確率 / 時間 (effectiveness ratio) | **(A') B1-lite** (0.26 / 17.5h ≈ 1.5%/h) > (C) Q-aware (0.15 / 3h ≈ 5.0%/h) — **(C) が単独最強だが分散化のみ** |

```sql
SELECT confirmed_order FROM horse_races
 WHERE jockey_code = ? AND track_code = ?
   AND (race_year || race_month_day) < ?   -- ★ strict less-than, current race を含まない
   AND confirmed_order > 0                  -- ★ DNF/出走取消除外
 ORDER BY (race_year || race_month_day) DESC
 LIMIT ?
```

**新 features は全て**:
- `(race_year || race_month_day) < before_date` 厳密で current race を物理的に除外
- 当日公表 fields (going, post-time horse weight, waku) は predict time access OK だが、**docstring に「予測時 access source: <field名>」を明記**
- corner_order_4 は post-race 確定値、past 走分のみ使用、当日 access 禁止

### 4.2 B1-S0 pre-flight (= H3-1/2/3 を統合): scope 絞り込み計画

**所要**: 3-4h (= v4 §5 H3 + 設計時間)

**B1-S0-step1**: H3-1 MING dynamics changelog 検証 (0.5h)
- `jvlink_client/ingest.py` + `predictor/features.py` で 2025-2026 間 MING fields 取り扱いに変更があったか git log で確認
- 変更があれば B1 scope に「MING 互換性 layer 追加」を追加検討

**B1-S0-step2**: H3-2 同季節 Brier 悪化 statistical test (0.5h)
- v4 N1 副指標 +3.3σ (Apr-May 2025 vs 2026) の Welch t-test または monthly bootstrap CI
- p < 0.05 で有意 = drift 仮説 確定 → B1 で season-aware calibration を必須に
- p >= 0.05 で非有意 = small sample で判定不能 → B1 で season-aware calibration は **option** 扱い

**B1-S0-step3**: H3-3 2026Q1 LGBM 崩壊 root cause 探索 (2-3h)
- 2026Q1 picks (276 picks per v4 §1.1) を以下軸で分解:
  - 馬場状態 (going 1-4) × hit_rate
  - 出走頭数 (head_count) × hit_rate  
  - 新馬比率 (= 当該レースの「新馬」フラグ持ちの馬の割合) × hit_rate
  - グレード × hit_rate
  - 距離帯 × hit_rate
- 各軸で 2026Q1 vs 2025Q1 比較、anomaly が >2σ なら causal candidate
- causal candidate に対応する Tier 2/3 features を選定

**B1-S0 → B1-S1 gate**:
- H3-3 で causal candidate >= 1 個特定 → B1-S1 着手 (= Tier 2/3 から 2-3 features 選定)
- causal candidate 0 個 → B1-S1 着手保留、別経路 (= §2 (B) calibration-only fix or (C) operational hedge) を検討
- causal candidate 多数 (>=4 個) → scope 絞り込み困難、B1-S1 で feature ablation 実験を中心に再設計
