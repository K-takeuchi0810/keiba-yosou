## 総合: 4.4 / 5  (前回 4.2 → +0.2)

P0-3 で **重賞ホワイトリストモード + `whitelist_only_*` 集計を backtest 出力に新設**。改修と同期で 2 件の長期 backtest が `data/backtest/` に保存され、`p03-whitelist-on` (集計追加前) と `p03-whitelist-with-only` (集計追加後) で **rule_version 単位の段階比較が時系列に残った**。1,164 戦・4 ヶ月の長期窓で `all=60.8%` / `buy_only=0 戦 (発火せず)` / **`whitelist_only=326 戦 / 84.9%`** という 3 系統並列出力が確立し、前回 scorecard で「3 つの新事実」として指摘した「graded 黒字」が **専用フィルタ + 専用集計列** として運用パイプラインに組み込まれた。`whitelist_only_return_rate=0.849` は控除率 (単勝 80%) を依然 5 ポイント下回るが、**1,164 戦中 326 戦という有意なサンプルサイズで 84.9% に達したのは過去最高**で、初めて「黒字に手が届く対象集合」を backtest が特定できた。

## 項目別

- **バックテスト設計の正しさ: 5/5 (4 → 5, +1)** — `all_*` / `buy_only_*` / `whitelist_only_*` の **3 系統並列出力** が完成。同一 1,164 戦に対し 3 つの異なる買い方を 1 ファイル内で同時計測できる構造になり、フィルタ A/B 比較の自由度が一段上がった。`buy_only_bets=0` を保ったまま `whitelist_only_bets=326` で発火、`races_total / races_no_horses / races_no_pick / races_filtered / races_tentative_skipped / races_bet` の遷移カウンタも従来通り維持。設計面で「同種アプリ上位」と言える状態に到達。
- **時系列リーク防止: 4/5 (±0)** — 改修対象外。calibrator は依然 16,550 件全部学習・全部評価の自己参照状態。次回の walk-forward 化で 5 点候補。
- **calibration / reliability 計測: 5/5 (±0)** — 前回 5 点維持。Brier=0.0575 / LogLoss=0.2098 / 12 bin reliability テーブルが 2 ファイルとも保存され、whitelist 改修が calibration 数値に影響を与えていないことが時系列で確認できる (= 重賞限定モードは予測値そのものを変えず、買い目選択だけを変える設計が backtest で実証)。
- **A/B 比較 / バージョン管理: 5/5 (4 → 5, +1)** — `p03-whitelist-on` と `p03-whitelist-with-only` の **2 ステップ rule_version** がファイル名 + JSON 内部に明記され、**「改修内の小ステップ単位の差分」も追跡可能** に。前回 1 点減点理由 (rule_version 強制 bump の機械的ガード無し) は構造的に残るが、今回の 2 件保存により **「改修者は実際に rule_version を bump する文化が定着した」** ことが過去 4 連続 backtest (p02 → p03-on → p03-with-only) で実証されたため、運用面で 5 点へ繰り上げ。
- **過適合監視 / 期間分割評価: 3/5 (4 → 3, -1)** — **減点**。前回大幅改善した項目だが、`whitelist_only=84.9%` という強い結果が出た以上、**「2026/01–04 で whitelist 設定 (graded 限定) を決め、同じ 2026/01–04 で 84.9% を計測している」自己参照リスク** が顕在化。重賞は元々サンプルが薄く (326 戦) ボラが大きいため、**ホールドアウト無しでこの数値を信じて運用するのは危険**。walk-forward が無い構造的問題が **今回の改修で初めて現実の実害リスクとして表面化** したので、機械的に 1 点減点。

## 主な改善提案 (優先 1 件、最大 3 件)

1. **whitelist_only=84.9% のホールドアウト検証** — `scripts/backtest.py` に `--holdout-from 20260301` を追加し、2026/01–02 (約 600 戦) で whitelist 構成を凍結 → 2026/03–04 (約 560 戦) でホールドアウト評価。326 戦の 84.9% が「期間内サンプル選択バイアスで膨らんでいるだけ」なのか「真に黒字に手が届く構造」なのかを **次回改修の前に必ず分離**。これをやらずに graded 限定を本番運用すると、ホールドアウトで 60% 台に戻ったとき責任の所在が消える。期待効果: 84.9% の信頼区間が初めて機械的に出る。
2. **whitelist 内部の構成分解** — `whitelist_only_*` をさらに `by_class` (G1/G2/G3/L) でブレイクダウン。326 戦の内訳が G3 偏重なのか G1/G2 でも黒字なのかが不明なまま。`scripts/backtest.py` の集計部に 5 行追加。期待効果: ホールドアウトで 60% 台に戻ったとしても、サブセットで黒字維持される層が見える可能性 → さらに絞った v2 whitelist の根拠データになる。
3. **buy_filter が 1,164 戦 0 発火問題への対処** — 前回提案で指摘済みだが今回も `buy_only_bets=0` のまま。whitelist 系統が機能した以上、buy_filter 系統は **死に系統** として `config.BUY_FILTER_DEFAULT` の閾値を 1,164 戦のうち 100–300 戦は採用される水準まで緩めるか、buy_only を廃止して whitelist_only に一本化するかを判断すべき時期。死に系統を放置すると JSON スキーマだけ重くなる。
